# Family Chat — Project Overview

## 1. Purpose and Scope

Family Chat — это веб‑чат для одного родителя (администратора) и нескольких детей, предназначенный для безопасного и простого общения внутри семьи.

Цель проекта — обкатать «правильный» инженерный и multi‑agent подход к разработке (архитектура, процессы, документация), сохраняя при этом простоту функционала.

В текущий scope входят базовые возможности текстового чата в браузере с одним общим каналом для всех участников.

## 2. Users and Roles

- Родитель (Administrator)
  - Создаёт и управляет аккаунтами детей (логин/пароль).
  - Может входить в общий чат и участвовать в переписке.
- Ребёнок (User)
  - Использует выданную пару логин/пароль для входа.
  - Участвует в общем чате.

Возрастной диапазон детей: от 11 лет. Интерфейс, формулировки и визуальный стиль должны быть достаточно простыми и понятными для этого возраста.

## 3. Functional Requirements

### 3.1 Аутентификация

- Логин:
  - Пользователь вводит логин и пароль.
  - При успешной проверке попадает в основной экран чата.
  - При ошибке — видит понятное сообщение о неправильных данных.
- Регистрация:
  - Самостоятельная регистрация пользователей отсутствует.
  - Учетные записи (логин/пароль) создаются только родителем‑администратором.

### 3.2 Управление пользователями (родитель)

- Родитель может:
  - Создавать нового пользователя‑ребёнка с парой логин/пароль.
  - Просматривать список существующих пользователей.
  - Изменять пароль пользователя.
  - Деактивировать пользователя (опционально — можно оставить как future).

### 3.3 Чат

- Общий чат:
  - На текущем этапе существует один общий чат‑комната, доступный всем пользователям.
  - Все участники видят одинаковую ленту сообщений.
- Отправка сообщений:
  - Пользователь вводит текст до 150 символов (включая эмодзи).
  - При отправке сообщение сохраняется в БД и транслируется всем подключённым пользователям.
- Получение сообщений:
  - Пользователь видит новые сообщения в общем чате в режиме близком к реальному времени.
- История:
  - При входе в чат пользователь видит историю сообщений за последние 30 дней.
  - Сообщения старше 30 дней регулярно удаляются (retention job).

### 3.4 Эмодзи

- Пользователь может отправлять сообщения, содержащие эмодзи.
- Конкретный поддерживаемый список (набор эмодзи) будет определён отдельно.

### 3.5 Антиспам

- Один пользователь не может отправлять более 1 сообщения в секунду.
- При превышении лимита отправка блокируется и показывается понятное сообщение об ограничении.

### 3.7 Чат-бот Lulu

- В общем чате присутствует бот Lulu — виртуальный участник, отвечающий на вопросы пользователей.
- Обращение к боту:
  - Пользователь пишет сообщение, содержащее `@Lulu` (имя бота настраивается в конфигурации).
  - Регистрозависимо: `@lulu` не активирует бота.
- Ответ бота:
  - Lulu отвечает в общий чат — ответ виден всем участникам.
  - Сообщение выглядит как обычное сообщение пользователя с иконкой бота.
  - Ответ сохраняется в БД наравне с обычными сообщениями (попадает в историю).
- Кто может обращаться: все пользователи (родитель и дети).
- Ограничения:
  - Применяется тот же антиспам-лимит: не более 1 сообщения в секунду от пользователя.
  - Нельзя создать пользователя с именем бота через User Management.
- Источник ответов: OpenAI API (без контентных ограничений на уровне приложения).
- При недоступности OpenAI:
  - Lulu отправляет в чат локализованное сообщение об ошибке.
- Имя бота задаётся в конфигурации (`BOT_NAME` в `.env`) и может быть изменено без изменения кода.

### 3.6 Обработка ошибок

- Если сервис недоступен, пользователь видит дружелюбное сообщение об ошибке
  (например: «Сервис временно недоступен, попробуйте позже»).
- В случае ошибки отправки конкретного сообщения пользователь явно видит, что сообщение не было доставлено
  (статус или уведомление).

## 4. Non‑Functional Requirements

### 4.1 Performance

- При 100 параллельных активных пользователях задержка доставки сообщения
  (от момента отправки до отображения у других пользователей) не должна превышать 1 секунду
  в условиях сети 100 Мбит/с.
- Максимальный размер одного сообщения: 150 символов.
- Система должна корректно работать при количестве сообщений в БД до 10 000.

### 4.2 Availability and Reliability

- Целевая доступность: 99 %.
- Правило для сообщений:
  - Сообщение либо успешно сохраняется и доставляется всем участникам, либо пользователь явно видит ошибку отправки.
- Определение «сервис не работает»:
  - API систематически возвращает 5xx‑коды, **или**
  - WebSocket‑соединение не устанавливается или нестабильно рвётся, **или**
  - База данных недоступна (ошибки подключения / критические ошибки запросов).
- В случае недоступности сервис должен:
  - Блокировать отправку новых сообщений.
  - Показывать пользователю дружелюбное сообщение о недоступности.

### 4.3 Degradation Behavior

- Частичная деградация:
  - Если недоступна отправка сообщений, но история доступна:
    - Пользователь может читать историю.
    - Кнопка отправки деактивируется или показывает соответствующее сообщение.
  - Если недоступна история, но отправка работает:
    - Пользователь видит сообщение о временной недоступности истории.
    - Текущая переписка (отправка/получение новых сообщений) продолжает работать.
  - При невозможности корректно отработать частичную деградацию допускается переход в режим
    «сервис недоступен» с единым сообщением об ошибке.

### 4.4 Security and Privacy

- На текущем этапе шифрование транспорта не используется (нет HTTPS/WSS).
- Архитектура должна позволять добавить шифрование позднее без изменения бизнес‑логики.
- Все данные пользователей и сообщений хранятся в БД; прямой доступ к БД имеют только серверные компоненты.

### 4.5 Architecture and Scalability

- Архитектура закладывается как микросервисная и масштабируемая:
  - Логическое разделение как минимум на:
    - Auth/Users Service
    - Chat/Messages Service
    - (опционально) Gateway/WebSocket Service
  - Сервисы должны быть спроектированы так, чтобы их можно было масштабировать горизонтально
    (состояние вынесено в БД/кэш).
- В дальнейшем планируется:
  - Добавление новых ролей (например, дополнительные взрослые / модераторы).
  - Поддержка отдельных комнат и приватных бесед.
- Текущее решение должно минимизировать необходимость ломки схемы данных и протоколов
  при добавлении этих возможностей.

### 4.6 Code Quality and Testing

- Core‑функциональность (аутентификация, авторизация, отправка/получение сообщений,
  запись в БД, базовая валидация) — не менее 80 % покрытия автоматическими тестами.
- Некритичные части (неосновные UI‑элементы, вспомогательные функции) — не менее 50 % покрытия.
- Типы тестов:
  - Unit‑тесты для бизнес‑логики.
  - Интеграционные тесты для работы с БД и основными API.

### 4.7 Logging and Observability

- Логгируются только ошибки (ошибки БД, неудачные попытки аутентификации, сбои отправки сообщений,
  исключения в сервисах).
- Метрики, трассировка и алёртинг на данном этапе не реализуются, но архитектура должна
  позволять добавить их позднее без радикальной переработки.

### 4.8 Environment and Platform

- Клиент:
  - Веб‑браузер: Google Chrome последней стабильной версии (desktop).
  - Мобильная оптимизация и отдельные приложения (iOS/Android/desktop) отсутствуют.
- Сеть:
  - Требуемая пропускная способность: 100 Мбит/с для достижения целевой задержки.
- Деплой:
  - Сервис развёртывается на бесплатном облачном сервисе (конкретный провайдер будет выбран отдельно).

### 4.9 Localization

- Интерфейс должен поддерживать как минимум русский и английский языки.
- Допускается реализация в виде простого переключения RU/EN без полноценной i18n‑инфраструктуры,
  но архитектура должна позволять расширение списка локалей.

## 5. Out of Scope

В текущем релизе **не входят**:

- Самостоятельная регистрация пользователей.
- Шифрование трафика (HTTPS/WSS).
- Пуш‑уведомления и нативные мобильные/desktop‑приложения.
- Голосовые и видео‑звонки.
- Отправка файлов, изображений и других вложений.
- Продвинутый антиспам (CAPTCHA, поведенческий анализ).
- SLA по времени восстановления сервиса.

## 6. Assumptions and Constraints

- Все пользователи — участники одной доверенной семейной группы; злонамеренные атаки и
  сложные сценарии безопасности не рассматриваются.
- Выбранный бесплатный облачный провайдер способен выдержать:
  - До 100 одновременных активных пользователей.
  - До 10 000 сообщений в БД без дополнительной оптимизации.
- Проект создаётся в первую очередь как учебный и демонстрационный для отработки процессов,
  архитектурных и multi‑agent практик.
