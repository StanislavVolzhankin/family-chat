# Workflow: Feature Implementation

Этот документ описывает, как фича проходит полный цикл разработки в проекте Family Chat с участием всех агентов.

Участники:

- Orchestrator / Team Lead Agent
- Architect Agent
- Backend Dev Agent
- Frontend Dev Agent
- DB / Schema Agent
- QA / Test Agent
- Reviewer / Watchdog Agent
- (опционально) DevOps / Deployment Agent

---

## 1. Вход: новая фича / изменение

**Источник:** человек (владелец проекта) или backlog.

**Формат задачи:**

- Краткое описание: что хотим добавить/изменить.
- Цель: зачем это нужно (с точки зрения пользователей/системы).
- Ссылки: на релевантные части `PROJECT_OVERVIEW.md` и `ARCHITECTURE.md` (если есть).
- Приоритет: насколько это важно относительно других задач (опционально).

---

## 2. Шаг Orchestrator: первичный разбор задачи

**Ответственный:** Orchestrator / Team Lead Agent

1. Читает описание задачи.
2. Сверяется с:
   - `PROJECT_OVERVIEW.md` (требования).
   - `ARCHITECTURE.md` (текущая архитектура).
3. Классифицирует задачу:
   - Новая фича / изменение существующей / багфикс.
4. Решает, нужен ли Architect Agent:
   - **Нужен**, если:
     - Затрагивается несколько компонент (backend + frontend + DB).
     - Есть влияние на архитектуру или модель данных.
     - Меняются нефункциональные требования.
   - **Не нужен**, если:
     - Мелкий, локальный UI‑фикс или подобное.

**Результат:** статус задачи → `planned` или запрос уточнений человеку.

---

## 3. Шаг Architect: план и архитектурные решения

**Ответственный:** Architect Agent (по запросу Orchestrator)

1. Получает от Orchestrator:
   - Описание задачи.
   - Контекст (ссылки на требования/архитектуру).
2. Анализирует:
   - Какие компоненты/модули затронуты.
   - Требуются ли изменения архитектуры или только реализация в рамках существующей.
3. Формирует:
   - Высокоуровневый план реализации.
   - Разбиение на подзадачи по агентам:
     - `backend: ...`
     - `frontend: ...`
     - `db/schema: ...` (если нужно)
     - `qa/tests: ...`
   - Необходимые изменения в документации (например, какие разделы `ARCHITECTURE.md` стоит обновить).
4. Возвращает план Orchestrator.

**Результат:** задача всё ещё в статусе `planned`, но теперь с архитектурным планом и списком подзадач.

---

## 4. Шаг Orchestrator: декомпозиция и назначение подзадач

**Ответственный:** Orchestrator / Team Lead Agent

1. На основе плана Architect (или самостоятельно для простых задач) формирует подзадачи:
   - Backend Dev — изменения в Laravel + PostgreSQL.
   - Frontend Dev — изменения в React.
   - DB / Schema — изменения в схеме/миграциях (если нужны).
   - QA / Test — тест‑план и тесты.
2. Ясно формулирует:
   - Цель подзадачи.
   - Ограничения (что нельзя ломать/менять).
   - Формат ожидаемого результата (патчи, список файлов, тесты).
3. Переводит задачу в статус `in_progress` и раздаёт подзадачи Dev‑агентам.

---

## 5. Шаг Dev‑агентов: реализация

### 5.1 Backend Dev Agent

- Реализует:
  - Изменения в API, WebSocket‑логике, фоновых задачах.
  - Бизнес‑правила (валидация, антиспам, обработка ошибок).
- Согласует изменения схемы с DB / Schema Agent (если нужно).
- Обновляет/пишет backend‑тесты (совместно с QA, если требуется).
- Возвращает Orchestrator:
  - Список изменений (что, где и как изменено).
  - Информацию о тестах (что покрыто).

### 5.2 Frontend Dev Agent

- Реализует:
  - Изменения в UI (компоненты React).
  - Логику работы с WebSocket (отправка, приём, reconnect).
  - Состояния ошибок, недоступности сервиса, блокировки отправки.
  - Обновления в локализации (RU/EN).
- Пишет/обновляет простые UI‑тесты (по договорённости).
- Возвращает Orchestrator:
  - Список изменений в компонентах и локализации.
  - Описание UX‑поведения.

### 5.3 DB / Schema Agent (по необходимости)

- Проектирует и описывает изменения схемы:
  - Новые таблицы/поля/индексы.
- Подготавливает описание миграций.
- Возвращает Orchestrator и Backend Dev:
  - Новую схему и рекомендации по её использованию.

**Результат:** после получения ответов от всех Dev‑агентов Orchestrator убеждается, что реализация этапа закончена, и переводит задачу в статус `ready_for_QA`.

---

## 6. Шаг QA: тест‑план и проверка

**Ответственный:** QA / Test Agent

1. Получает от Orchestrator:
   - Описание фичи и требования.
   - План Architect (если есть).
   - Изменения от Backend/Frontend/DB.
2. Разрабатывает:
   - Тест‑план: какие сценарии проверяются (happy‑path, ошибки, edge‑cases).
   - Список тест‑кейсов (включая негативные).
3. Реализует/предлагает автотесты:
   - Unit‑тесты для ключевой логики.
   - Интеграционные тесты для API/БД.
4. Оценивает:
   - Насколько покрыты core‑сценарии (с учётом цели 80 % core, 50 % non‑core).
5. Возвращает Orchestrator:
   - Результаты тестов (passed/failed).
   - Найденные дефекты (если есть).
   - Рекомендации.

Если найдены проблемы:

- Orchestrator возвращает соответствующие подзадачи Dev‑агентам на доработку.
- После исправлений QA повторяет критичные проверки.

При успешном результате Orchestrator переводит задачу в статус `ready_for_review`.

---

## 7. Шаг Reviewer: финальное ревью

**Ответственный:** Reviewer / Watchdog Agent

1. Получает от Orchestrator:
   - Описание задачи.
   - План Architect (если есть).
   - Патчи/описания изменений от Dev‑агентов.
   - Результаты QA и тест‑планы/кейсы.
2. Проверяет:
   - Соответствие требований (`PROJECT_OVERVIEW.md`).
   - Соответствие архитектуре (`ARCHITECTURE.md`).
   - Качество кода (читаемость, простота, отсутствие очевидных антипаттернов).
   - Потенциальные проблемы (перформанс, устойчивость к ошибкам, безопасность).
3. Выдаёт вердикт:
   - `approve` — изменений достаточно, можно принимать.
   - `needs_changes` — список конкретных замечаний.

Если нужны правки:

- Orchestrator возвращает задачи соответствующим Dev‑агентам (и при необходимости Architect).
- После исправлений ревью повторяется.

При `approve` Orchestrator переводит задачу в статус `done` (с точки зрения команды агентов).

---

## 8. Шаг Orchestrator: summary для человека

**Ответственный:** Orchestrator / Team Lead Agent

1. Составляет краткое summary для человека:
   - Что было сделано (по пунктам).
   - Какие файлы/модули изменены (backend, frontend, DB, тесты).
   - Какие тесты добавлены/обновлены и их статус.
2. Отмечает:
   - Остались ли важные TODO или улучшения на будущее.
3. Отдельно подчёркивает:
   - Что никакие изменения ещё не попали в основную ветку без твоего ручного апрува и merge.

---

## 9. Роль DevOps / Deployment Agent в воркфлоу (опционально)

Когда фича готова и approved:

- Orchestrator может создать отдельную задачу для DevOps / Deployment Agent:
  - Обновить инструкции по деплою для новой фичи (если требуется).  
  - Проверить, нужно ли менять конфигурацию окружения.  
  - Актуализировать CI/CD (например, добавить новый шаг для тестов).

DevOps Agent:

- Предлагает изменения в конфиги/пайплайны (без прямого редактирования кода).  
- Обновляет/расширяет документацию по деплою.

---

## 10. Инварианты безопасности и контроля

На протяжении всего workflow соблюдаются правила:

- Ни один агент не коммитит и не пушит изменения в основную ветку.  
- Все изменения проходят:
  - Dev‑агенты → QA → Reviewer → человек (владелец репозитория).
- Архитектурные решения принимаются через Architect Agent, а не отдельными Dev‑агентами.
- Orchestrator отвечает за то, чтобы последовательность шагов не нарушалась.

Этот workflow‑документ задаёт «стандартный процесс» для реализации фич в проекте Family Chat.
