# Family Chat — Architecture

## 1. Overview

Family Chat — веб‑приложение для семейного текстового общения (один родитель‑администратор и несколько детей), реализованное на стеке:

- Backend: PHP (Laravel), модульный монолит.
- Frontend: React SPA.
- Database: PostgreSQL.
- Realtime: WebSocket для доставки новых сообщений, HTTP API для аутентификации и загрузки истории.

Архитектура строится как модульный монолит с явными доменными модулями (Auth/Users, Chat/Messages, Bot, Localization), спроектированными так, чтобы в будущем их можно было выделить в отдельные микросервисы.

Цели архитектуры:

- Обеспечить простую, читаемую структуру для pet‑проекта.
- Заложить правильные границы модулей и расширяемость (роли, комнаты, бот, шифрование).
- Поддержать требования по производительности (1 сек при 100 параллельных пользователях) и качеству.

## 2. High-Level Components

### 2.1 Web Client (React)

Ответственность:

- Пользовательский интерфейс (RU/EN), управление состоянием приложения.
- Экран логина, экран чата, отображение истории сообщений.
- Взаимодействие с backend по HTTP (REST API) и по WebSocket.

Взаимодействие:

- HTTP:
  - POST /api/auth/login — логин.
  - GET /api/messages — загрузка истории сообщений.
  - (в будущем) API для управления пользователями (для родителя).
- WebSocket:
  - Подключение к WebSocket‑серверу для отправки и получения сообщений в реальном времени.
  - Обработка событий reconnect и ошибок.

### 2.2 API Backend (Laravel)

Ответственность:

- Обработка HTTP‑запросов от клиента.
- Реализация доменной логики модулей Auth/Users и Chat/Messages.
- Доступ к PostgreSQL через ORM/репозитории.
- Планировщик заданий (cron/scheduler) для обслуживания системы (retention сообщений).

Основные зоны:

- Auth/Users API:
  - Логин.
  - Управление пользователями (только для родителя).
- Chat API:
  - Загрузка истории сообщений.
- Service Layer:
  - Бизнес‑логика (валидация, правила, антиспам).
- Infrastructure:
  - Доступ к БД, взаимодействие с WebSocket‑слоем.

### 2.3 WebSocket Gateway

Ответственность:

- Обслуга WebSocket‑подключений клиентов.
- Приём команд send_message от клиентов.
- Рассылка событий new_message всем подключённым пользователям.

Особенности:

- Аутентификация по токену (передаётся при подключении).
- Валидация сообщений (антиспам, длина, доступность БД).
- Может быть реализован:
  - Либо встроенным компонентом/пакетом в рамках Laravel‑приложения.
  - Либо отдельным процессом, но в пределах одного deployment’а (по‑прежнему модульный монолит).

### 2.4 PostgreSQL Database

Ответственность:

- Хранение данных о пользователях и сообщениях.
- Поддержка выборок истории за последние 30 дней.
- Обеспечение индексов для эффективных запросов (по времени, по пользователю).

## 3. Domain Modules (Logical Services)

### 3.1 Auth / Users Module

Ответственность:

- Аутентификация пользователей по логин/пароль.
- Управление пользователями родителем (создание, изменение пароля, деактивация).

Сущность:

- User
  - id (PK)
  - username (unique)
  - password_hash
  - role (enum: parent, child)
  - is_active (bool)
  - created_at, updated_at

Основные операции (HTTP API):

- POST /api/auth/login
  - Вход: username, password.
  - Выход: токен (например, JWT или session id), role, базовая информация о пользователе.
- POST /api/users (только parent)
  - Создание нового пользователя‑ребёнка (username, password).
- GET /api/users (только parent)
  - Список пользователей (для простого менеджмента).
- PATCH /api/users/{id} (только parent)
  - Смена пароля, переключение is_active.

Правила:

- Только один пользователь с ролью parent.
- Дети не могут создавать/менять пользователей.

### 3.2 Chat / Messages Module

Ответственность:

- Управление общим чатом.
- Сохранение новых сообщений и выдача истории.
- Применение антиспам‑правил.

Сущность:

- Message
  - id (PK)
  - user_id (FK → users.id)
  - content (varchar(150))
  - created_at (timestamp)

Основные операции:

- HTTP:
  - GET /api/messages?limit=N&since=timestamp
    - Возвращает сообщения за последние 30 дней, отсортированные по created_at.
- WebSocket:
  - Client → Server:
    - send_message { content }
  - Server → Clients:
    - new_message { id, user_id, username, content, created_at }
  - Server → Client (ошибки):
    - error { code, message }

Правила:

- Максимальная длина content — 150 символов (включая эмодзи).
- Антиспам: один пользователь не может отправить более 1 сообщения в секунду.
- Retention: сообщения старше 30 дней удаляются ежесуточным job’ом.

### 3.4 Bot Module

Ответственность:

- Обнаружение обращений к боту в сообщениях (`@{BOT_NAME}`).
- Отправка запроса в OpenAI API и получение ответа.
- Сохранение ответа бота как сообщения в БД и broadcast всем участникам чата.
- Обработка ошибок недоступности OpenAI (локализованное сообщение об ошибке в чат).

Конфигурация:

- `BOT_NAME` — имя бота (например, `Lulu`), задаётся в `.env`, хранится в `config/bot.php`.
- `OPENAI_API_KEY` — ключ OpenAI API, задаётся в `.env`.
- `OPENAI_MODEL` — модель (например, `gpt-4o-mini`), задаётся в `.env`.

Сущность Lulu в БД:

- Lulu хранится как специальная запись в таблице `users` (создаётся при первом запуске / seeder).
- Поле `is_bot = true` идентифицирует бота.
- `password_hash` — случайная строка (логин невозможен).
- `role = child`, `is_active = true`.
- Username совпадает с `BOT_NAME` из конфига.

Правила:

- Нельзя создать пользователя с именем бота через User Management API.
- Антиспам применяется к пользователю, отправившему запрос (не к ответам бота).
- Лимит длины ответа бота: 150 символов (обрезается при необходимости).
- Ответ бота сохраняется в `messages` с `user_id = Lulu.id`.

Флоу обработки:

1. WebSocket handler получает `send_message { content }` от пользователя.
2. Сохраняет сообщение пользователя в БД, рассылает всем (стандартный флоу).
3. `BotService::detect($content)` проверяет наличие `@{BOT_NAME}`.
4. Если обнаружено — асинхронно вызывает OpenAI API с вопросом пользователя.
5. Ответ сохраняется как сообщение от Lulu, рассылается всем как `new_message`.
6. При ошибке OpenAI — Lulu отправляет локализованное сообщение об ошибке.

### 3.3 Localization Module

Ответственность:

- Хранение и предоставление локализованных строк (RU/EN).
- Поддержка переключения языка.

Реализация (минимальная):

- Набор словарей на фронтенде:
  - /locales/ru.json
  - /locales/en.json
- Возможная базовая локализация текстов ошибок/валидации на backend.

## 4. Client–Server Interaction

### 4.1 Login Flow

1. Пользователь открывает React‑приложение.
2. Вводит логин и пароль.
3. Клиент отправляет запрос:
   - POST /api/auth/login { username, password }.
4. Backend:
   - Валидирует данные.
   - Находит пользователя в БД.
   - Проверяет пароль.
   - Возвращает токен (JWT/session id), роль пользователя (parent/child), статус успеха/ошибки.
5. Клиент:
   - Сохраняет токен.
   - Переключает интерфейс на экран чата.
   - Инициирует запрос истории сообщений и подключение к WebSocket.

### 4.2 History Load Flow

1. После успешного логина клиент отправляет:
   - GET /api/messages?limit=N (и опционально since=timestamp).
2. Backend:
   - Выбирает сообщения за последние 30 дней из БД (сортировка по created_at).
   - Ограничивает количество (limit).
3. Клиент:
   - Отображает историю в UI.

### 4.3 WebSocket Connection Flow

1. Клиент открывает WebSocket‑подключение:
   - ws://<host>/ws/chat?token=<access-token> (или токен в header).
2. WebSocket Gateway:
   - Валидирует токен через Auth‑модуль.
   - При успехе регистрирует соединение (user_id ↔ socket_id).
   - При ошибке аутентификации закрывает соединение, клиент получает сообщение о необходимости повторного логина.

### 4.4 Message Send / Receive Flow

- Отправка:

  1. Пользователь вводит текст и нажимает «Отправить».
  2. Клиент отправляет по WebSocket:
     - send_message { content }.
  3. WebSocket Gateway / Chat‑модуль:
     - Проверяет аутентификацию пользователя (по соединению).
     - Проверяет антиспам‑лимит (не более 1 сообщения/сек).
     - Проверяет длину сообщения (≤150 символов).
     - Пытается сохранить Message в БД.
     - При успехе:
       - Рассылает всем активным соединениям событие:
         - new_message { id, user_id, username, content, created_at }.
     - При ошибке (валидация, БД, антиспам):
       - Отправляет текущему клиенту:
         - error { code, message }.

- Получение:

  1. Клиент слушает события new_message по WebSocket.
  2. При получении добавляет сообщение в локальный state и отображает его в UI.

## 5. WebSocket Reconnect and Error Handling

### 5.1 Поведение при обрыве соединения

- При разрыве WebSocket:

  - Клиент автоматически пытается переподключиться:
    - Используется экспоненциальный backoff (например: 0.5s, 1s, 2s, 4s, … до максимального интервала).
    - Счётчик попыток ограничен (например, 10 попыток).
  - В период отсутствия соединения:
    - Статус в UI: «соединение потеряно, пытаемся переподключиться…».
    - Кнопка отправки сообщений заблокирована.
    - Попытки отправки не ставятся в очередь, сразу показывается сообщение, что отправка невозможна без соединения.

- При успешном reconnect:

  - Счётчик попыток сбрасывается.
  - UI возвращается в «online» состояние.
  - При необходимости клиент может запросить обновлённую историю:
    - GET /api/messages?since=lastMessageTimestamp.

- При исчерпании попыток reconnect:

  - Клиент показывает состояние:
    - «чат недоступен, попробуйте позже / обновите страницу».
  - Отправка остаётся заблокированной.

### 5.2 Ошибки аутентификации

- Если при подключении или reconnect сервер возвращает ошибку аутентификации (недействительный токен):

  - Соединение закрывается.
  - Клиент:
    - Очищает локальный токен.
    - Переводит пользователя на экран логина с сообщением о необходимости войти заново.

## 6. Data Model (Draft)

### 6.1 Users Table

- id (PK, integer/bigint, auto‑increment)
- username (varchar, unique)
- password_hash (varchar)
- role (enum/string: parent, child)
- is_active (boolean)
- is_bot (boolean, default false)
- created_at (timestamp)
- updated_at (timestamp)

Индексы:

- UNIQUE(username)
- INDEX(role)
- INDEX(is_active)
- INDEX(is_bot)

### 6.2 Messages Table

- id (PK, integer/bigint, auto‑increment)
- user_id (FK → users.id)
- content (varchar(150))
- created_at (timestamp)

Индексы:

- INDEX(created_at)
- INDEX(user_id)

## 7. Retention and Maintenance

### 7.1 Retention Policy

- Сообщения хранятся в БД не более 30 календарных дней.
- Поведение:
  - При загрузке истории (GET /api/messages) возвращаются только сообщения за последние 30 дней.
  - Сообщения старше 30 дней не учитываются и постепенно удаляются.

### 7.2 Очистка старых сообщений

- В Laravel реализуется job/console command, которая:

  - Раз в сутки (например, ночью) запускается планировщиком.
  - Удаляет все записи из таблицы messages, у которых created_at старше 30 дней на момент запуска job’а.
  - Работает пакетами (batch), чтобы не перегружать БД.

- Особенности:

  - Очистка не требует даунтайма; новые сообщения продолжают сохраняться во время работы job’а.
  - Точность 30‑дневного периода — до суток (зависит от времени запуска job’а).

## 8. Deployment Overview

- Backend (Laravel API + WebSocket компонент) и PostgreSQL размещаются на **Fly.io** (бесплатный tier, Docker-based).
- Frontend (React build) деплоится как статический сайт на **Vercel** (бесплатно).
- Деплой построен на Docker: backend упакован в контейнер, конфигурация через `fly.toml`.
- В первой итерации:
  - Все backend‑модули (Auth/Users, Chat/Messages, Localization, WebSocket‑gateway) живут в одном Laravel‑приложении (модульный монолит).
- В дальнейшем возможно:
  - Выделение Auth/Users и Chat/Messages в отдельные сервисы.
  - Вынесение WebSocket‑gateway в отдельный сервис.
